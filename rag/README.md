# 🧠 RAG על תיעוד של כלי Agentic Coding

כלי Agentic Coding — Claude Code, Cursor, Kiro, Firebase Studio — מנהלים תיעוד שוטף של העבודה שלהם בקבצי md: הנחיות, החלטות, מפרטים ואזהרות. כל כלי שומר בתיקיות משלו, ואף אחד לא קורא את התיעוד של השני. הפרויקט הזה סורק את **כל** קבצי ה-md של כל הכלים בריפו, מאגד אותם לשכבת ידע אחת, ומאפשר לתשאל אותה בצ'אט: לענות על שאלות תוכן, להפיק רשימות מלאות, ולסנן לפי זמן ועדכניות.

הקורפוס כאן הוא התיעוד האמיתי של אפליקציית המתכונים שבריפו הזה:

| כלי | קבצים |
|---|---|
| Claude Code | `CLAUDE.md` |
| Cursor | `.cursor/rules/*.md`, `.cursor/notes/*.md` |
| Kiro | `.kiro/steering/*.md`, `.kiro/specs/**/*.md` |
| Firebase Studio | `docs/blueprint.md` |

## 🏗️ ארכיטקטורה

הפרויקט ממומש בשלושה שלבים, כולם חיים יחד בקוד:

**שלב א' — MVP סמנטי:** טעינה (`SimpleDirectoryReader` + מטא-דאטה של כלי/קובץ/תאריך/hash) ← חיתוך (`MarkdownNodeParser` לפי מבנה הכותרות, ואז `SentenceSplitter` לתחימת גודל) ← Embeddings של Cohere (`embed-multilingual-v3.0` — הקורפוס דו-לשוני, המודל האנגלי לא מספיק) ← אינדוקס ב-Pinecone דרך `VectorStoreIndex`.

**שלב ב' — Workflow מונחה-אירועים:** כל הצנרת בנויה כ-`Workflow` של LlamaIndex (`ragapp/workflow.py`). כל Step מקבל Event ומחזיר Event שקובע את הצעד הבא: ולידציית קלט ← ניתוב ← אחזור עם בדיקת ביטחון וניסיון-חוזר מורחב ← סינתזה. ראו [תרשים הזרימה](docs/workflow-diagram.html).

**שלב ג' — Data Extraction + Router:** ‏`extract.py` מחלץ מהקורפוס פריטים מובנים בעזרת Structured Prediction — **החלטות, כללים, אזהרות, שינויים** — כל אחד עם מקור (כלי + קובץ + כותרת) ותאריך `observed_at`, ושומר ל-`data/extracted_knowledge.json`. בזמן שאלה, `LLMSingleSelector` מנתב: שאלות תוכן ← חיפוש סמנטי; שאלות רשימה / עדכניות / טווח-זמן ← ה-LLM בונה `StructuredQuerySpec` (סוגי פריטים, מילות מפתח, טווח תאריכים, latest_only) שרץ על ה-JSON, והתוצאות נשלחות ל-LLM לניסוח תשובה.

### מבנה הקוד

```
rag/
├── prepare.py                  # שלב א': טעינה → חיתוך → embedding → Pinecone
├── extract.py                  # שלב ג': חילוץ ידע מובנה → data/extracted_knowledge.json
├── app.py                      # ממשק צ'אט Gradio (+ מצב --ask לטרמינל)
├── docs/workflow-diagram.html  # תרשים זרימה של ה-Workflow
└── ragapp/
    ├── config.py               # הגדרות, ספים, משתני סביבה
    ├── corpus.py               # מיפוי כלי→קבצים, טעינה עם מטא-דאטה
    ├── models.py               # Cohere (embeddings + LLM), Pinecone
    ├── schema.py               # סכמת Pydantic: פריטים, שאילתה מובנית, ניתוב
    ├── extraction.py           # חילוץ מובנה לכל קובץ, עם fallback ל-JSON
    ├── structured_store.py     # שאילתות על מאגר הידע (סוג/תגיות/תאריכים/עדכניות)
    ├── workflow.py             # ה-Workflow מונחה-האירועים (הלב של המערכת)
    └── llm_utils.py            # structured_predict עמיד עם fallback
```

## 🚀 התקנה והרצה

דרישות: Python 3.10+, מפתח [Cohere](https://dashboard.cohere.com/api-keys), מפתח [Pinecone](https://app.pinecone.io) (אין צורך ליצור אינדקס ידנית — `prepare.py` יוצר אותו).

```bash
cd rag
python -m venv .venv && source .venv/bin/activate   # Windows: .venv\Scripts\activate
pip install -e .

cp .env.example .env       # ולמלא את המפתחות

python prepare.py          # אינדוקס הקורפוס ל-Pinecone (מנקה namespace קודם)
python extract.py          # בניית מאגר הידע המובנה (שלב ג')
python app.py              # ממשק הצ'אט → http://127.0.0.1:7860
python app.py --ask "מה הצבע העיקרי של המערכת?"   # שאלה בודדת בטרמינל
```

> 🛜 בסביבת NetFree: ‏`pip install netfree-unstrict-ssl` — הקוד מפעיל אותו אוטומטית אם הוא מותקן.

> קבצי ה-md משתנים תוך כדי עבודה; המערכת מניחה תוכן סטטי — אחרי שינוי בקבצים מריצים שוב `prepare.py` ו-`extract.py` (ההרצה אידמפוטנטית).

## 💡 דיון — החלטות התכנון של שלב ב'

**איך מחלקים ל-Steps נכון?** לפי גבולות של החלטה, לא לפי גבולות של קוד: Step מסתיים בנקודה שבה התוצאה שלו יכולה לשנות את ההמשך (הקלט תקין? לאן לנתב? האם האחזור מספיק טוב?). חיתוך + embedding נשארו יחד ב-`prepare.py` כי אין ביניהם שום הסתעפות.

**האם Step חייב לכלול קריאת LLM?** לא. `validate_query` הוא בדיקות מחרוזת בלבד, ו-`retrieve` הוא וקטורי בלבד (ה-LLM נכנס רק אם צריך ניסוח-מחדש). Steps זולים בלי LLM הם בדיוק מה שמונע לשרוף טוקנים על קלט שממילא פסול.

**האם צריך State?** כן, אבל מינימלי. מונה הניסיונות-החוזרים חייב לשרוד בין הפעלות של אותו Step (אחרת לולאה אינסופית), והשאלה המקורית נחוצה לסינתזה גם אחרי שהאחזור רץ עם שאילתה מנוסחת-מחדש.

**מה שמים ב-Event ומה ב-State?** ב-Event — מה שהצעד הבא צריך כדי לרוץ (השאילתה הנוכחית, top_k, ה-nodes שנמצאו). ב-State (‏`Context.store`) — מה שחוצה צעדים: `original_query`, ‏`retries`, ‏`route`. כלל אצבע: אם מחר יתווסף Step בין שניים קיימים, מה שב-Event לא יישבר; מה שב-State זמין לכולם בלי להעביר אותו הלאה ביד.

## 🔍 רפלקציה

* **על מה ה-Agent עדיין לא עונה טוב?** שאלות שדורשות הצלבה כמותית בין מסלולים ("כמה מהאזהרות נוגעות להחלטות מהחודש האחרון?") — הניתוב בוחר מסלול אחד. ושאלות על מה ש**לא** כתוב ("על מה אין תיעוד?").
* **מתי הניתוב מפשל?** שאלות רשימה מנוסחות רך ("ספר לי על ההחלטות...") נשלחות לפעמים למסלול הסמנטי ואז הרשימה חלקית. ה-fallback (אין תוצאות מובנות ← סמנטי) מכסה את הכיוון ההפוך, אבל אין fallback מסמנטי למובנה.
* **שינוי קטן עם השפעה גדולה:** החלפת מודל ה-embedding מ-`embed-english-v3.0` (שב-Starter) ל-`embed-multilingual-v3.0` — על קורפוס עברי זה ההבדל בין אחזור אקראי לאחזור מדויק. גם הוספת הדרישה "מילות מפתח גם בעברית וגם באנגלית" ל-prompt של ה-spec שיפרה מאוד את המסלול המובנה.
* **מה הייתי משדרגת?** סנכרון אינקרמנטלי לפי hash הקובץ (המטא-דאטה כבר נשמר) במקום re-index מלא; מסלול "היברידי" שמריץ את שני החיפושים ומאחד; ו-re-ranking עם Cohere Rerank מעל האחזור הווקטורי.
