# 🧠 RAG על תיעוד של כלי Agentic Coding

כלי Agentic Coding — Claude Code, Cursor, Kiro, Firebase Studio — מנהלים תיעוד שוטף של העבודה שלהם בקבצי md: הנחיות, החלטות, מפרטים ואזהרות. כל כלי שומר בתיקיות משלו, ואף אחד לא קורא את התיעוד של השני. הפרויקט הזה סורק את **כל** קבצי ה-md של כל הכלים בריפו, מאגד אותם לשכבת ידע אחת, ומאפשר לתשאל אותה בצ'אט: לענות על שאלות תוכן, להפיק רשימות מלאות, ולסנן לפי זמן ועדכניות.

## מבנה הריפו

הריפו מכיל שני חלקים נפרדים בכוונה:

```
.
├── CLAUDE.md, docs/blueprint.md, .cursor/, .kiro/   ← "פרויקט הכנה": קבצי md בלבד
│                                                        (בלי קוד רץ) שכמה כלי Agentic
│                                                        Coding כתבו על פרויקט לדוגמה —
│                                                        זה הקורפוס שה-RAG עובד עליו
└── rag/                                              ← פרויקט ה-RAG עצמו — נפרד לחלוטין,
                                                          קורא ומתשאל את הקבצים שלמעלה
```

| כלי | קבצים |
|---|---|
| Claude Code | `CLAUDE.md` |
| Cursor | `.cursor/rules/*.md`, `.cursor/notes/*.md` |
| Kiro | `.kiro/steering/*.md`, `.kiro/specs/**/*.md` |
| Firebase Studio | `docs/blueprint.md` |

## 🏗️ ארכיטקטורה

**שלב א' — MVP סמנטי:** טעינה (`SimpleDirectoryReader` + מטא-דאטה של כלי/קובץ/תאריך/hash) ← חיתוך (`MarkdownNodeParser` לפי מבנה הכותרות, ואז `SentenceSplitter` לתחימת גודל) ← Embeddings של Cohere (`embed-multilingual-v3.0` — הקורפוס דו-לשוני, המודל האנגלי לא מספיק) ← אינדוקס ב-Pinecone דרך `VectorStoreIndex`.

**שלב ב' — Workflow מונחה-אירועים:** כל הצנרת בנויה כ-`Workflow` של LlamaIndex (`rag/ragapp/workflow.py`). כל Step מקבל Event ומחזיר Event שקובע את הצעד הבא: ולידציית קלט ← ניתוב ← אחזור עם בדיקת ביטחון וניסיון-חוזר מורחב ← סינתזה. ראו [תרשים הזרימה](rag/docs/workflow-diagram.html).

**שלב ג' — Data Extraction + Router:** ‏`rag/extract.py` מחלץ מהקורפוס פריטים מובנים בעזרת Structured Prediction — **החלטות, כללים, אזהרות, שינויים** — כל אחד עם מקור (כלי + קובץ + כותרת) ותאריך `observed_at`, ושומר ל-`rag/data/extracted_knowledge.json`. בזמן שאלה, `LLMSingleSelector` מנתב: שאלות תוכן ← חיפוש סמנטי; שאלות רשימה / עדכניות / טווח-זמן ← ה-LLM בונה `StructuredQuerySpec` שרץ על ה-JSON, והתוצאות נשלחות ל-LLM לניסוח תשובה.

## 🚀 התקנה והרצה

דרישות: Python 3.10+, מפתח [Cohere](https://dashboard.cohere.com/api-keys), מפתח [Pinecone](https://app.pinecone.io).

```bash
cd rag
python -m venv .venv && source .venv/bin/activate   # Windows: .venv\Scripts\activate
pip install -e ".[dev]"

cp .env.example .env       # ולמלא את המפתחות — לעולם לא לדחוף את הקובץ הזה ל-git

python prepare.py          # אינדוקס הקורפוס ל-Pinecone
python extract.py          # בניית מאגר הידע המובנה (שלב ג')
python app.py               # ממשק הצ'אט → http://127.0.0.1:7860
python app.py --ask "מה הצבע העיקרי של המערכת?"   # שאלה בודדת בטרמינל
pytest tests/ -v             # 35 בדיקות, רצות אופליין (בלי מפתחות API)
```

> 🛜 בסביבת NetFree: ‏`pip install netfree-unstrict-ssl` — הקוד מפעיל אותו אוטומטית אם הוא מותקן.

## 💡 דיון — החלטות התכנון של שלב ב'

**איך מחלקים ל-Steps נכון?** לפי גבולות של החלטה, לא לפי גבולות של קוד: Step מסתיים בנקודה שבה התוצאה שלו יכולה לשנות את ההמשך.

**האם Step חייב לכלול קריאת LLM?** לא. `validate_query` הוא בדיקות מחרוזת בלבד, ו-`retrieve` הוא וקטורי בלבד.

**האם צריך State?** כן, מינימלי: מונה הניסיונות-החוזרים והשאלה המקורית, ב-`Context.store`.

**מה שמים ב-Event ומה ב-State?** ב-Event — מה שהצעד הבא צריך. ב-State — מה שחוצה צעדים.

## 🔍 רפלקציה

* **על מה ה-Agent עדיין לא עונה טוב?** שאלות שדורשות הצלבה כמותית בין מסלולים, ושאלות על מה ש**לא** כתוב.
* **מתי הניתוב מפשל?** שאלות רשימה מנוסחות רך נשלחות לפעמים למסלול הסמנטי.
* **שינוי קטן עם השפעה גדולה:** החלפת מודל ה-embedding ל-multilingual, וקביעת מילות-מפתח דו-לשוניות ב-router.
* **מה הייתי משדרגת?** סנכרון אינקרמנטלי לפי hash, מסלול היברידי, re-ranking.
